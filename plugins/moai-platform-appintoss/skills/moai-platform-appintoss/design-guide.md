# 앱인토스 디자인 가이드

이 문서는 앱인토스 미니앱 디자인을 위한 상세 가이드입니다.

---

## 디자인 개요

앱인토스 디자인은 두 가지 도구를 제공합니다:

**앱빌더 (App Builder)**
- TDS 컴포넌트를 조립해 쉽고 빠르게 화면을 만듭니다
- 별도 설치 없이 웹에서 바로 사용 가능

**피그마 (Figma)**
- 권장 컴포넌트를 활용해 디자인을 완성합니다
- TDS 라이브러리 연동 지원

---

## 미니앱 브랜딩 가이드

앱인토스 서비스가 토스 서비스와 혼동되지 않도록 브랜딩 요소를 구분합니다.

### 1. 브랜드 로고

**규격:**
- 600×600px 정사각형
- 모서리 라운딩 없이 샤프하게
- 배경 포함 필수 (라이트/다크 모드 모두 대응)

**설정 방법:**

1. 앱인토스 콘솔에 로고 파일 업로드
2. `granite.config.mts`에 참조 설정

```typescript
export default defineConfig({
  brand: {
    icon: 'path/to/logo.png', // 로고 경로
    displayName: '서비스명',
    primaryColor: '#3182F6',
  },
});
```

**노출 위치:**
- 모든 탭
- 혜택 탭
- 푸시 알림
- 알럿
- 네비게이션
- 브릿지

### 2. 브랜드 네임

**언어 규칙:**
- 한글 사용 원칙 (예: "토스" O, "Toss" X)
- 특수한 상황에서만 예외 허용

**설정:**

```typescript
brand: {
  displayName: '서비스명', // 한글 권장
}
```

### 3. 브랜드 컬러

**선택 우선순위:**
1. 기존 브랜드 컬러 사용
2. 없으면 로고에서 지배적 컬러 추출

**컬러 추출 도구:**
- [color-thief](https://lokeshdhakar.com/projects/color-thief/) 활용

**접근성:**
- 대비 기준 미달 시 시스템이 자동 보정
- 원래 의도 유지하면서 접근성 확보

**형식:**
- # 기호 포함 6자리 HEX 코드 (예: #3182F6)

```typescript
brand: {
  primaryColor: '#3182F6',
}
```

### 4. 네비게이션 바

앱인토스 전용 플로팅 컴포넌트입니다.

- 게임 서비스: `/checklist/app-game.html#_5-내비게이션-바` 참고
- 비게임 서비스: `/checklist/app-nongame.html#_4-내비게이션-바` 참고

### 5. 브릿지

토스 앱에서 앱인토스로 진입 시 표시되는 전환 컴포넌트입니다.

- 게임: `/checklist/app-game.html#_1-접속-브릿지-뷰`
- 비게임: `/checklist/app-nongame.html#_1-접속-브릿지-뷰`

### 6. 탭 바

**설정 규칙:**
- 선택 사항이지만 구현 시 토스 플로팅 디자인 준수 필수
- 탭 개수: 최소 2개, 최대 5개
- 커스텀 UI 시 표준 디자인 준수 필요

## 다크패턴 방지 정책

앱인토스 사용자는 모든 서비스에서 일관되고 신뢰할 수 있는 경험을 기대합니다.

### 핵심 원칙

"고객의 예상을 뛰어넘는 디자인은 앱인토스 서비스로 출시될 수 없습니다."

모든 서비스는 별도 학습 없이 직관적으로 사용 가능해야 합니다.

### 5가지 금지 패턴

#### 1. 서비스 진입 시 바텀시트 즉시 표시

**금지 사유:**
- 전면 광고 바텀시트
- 알림 동의 요청

예상치 못한 중단은 참여를 방해하고 이탈 위험을 높입니다.

#### 2. 뒤로가기 시 예상치 못한 바텀시트

**금지 사유:**
- 다른 서비스 탐색 시 알림 동의 다이얼로그 표시

자율성을 훼손하고 유지 조작처럼 느껴져 신뢰가 하락합니다.

#### 3. 종료 기능 부재

**금지 사유:**
- 파트너 CTA만 선택 가능하게 하고 대안 없음

강압적으로 느껴지며 불신과 부정적 감정을 유발합니다.

#### 4. 사용자 플로우 중 예상치 못한 광고

**금지 사유:**
- 아이템 수집이나 메뉴 선택 중 전면 광고

몰입을 방해하고 부정적 브랜드 연상을 만듭니다.

#### 5. 모호하거나 불명확한 CTA 버튼

**금지 사유:**
- 기존 화면 설명을 반복하는 CTA

사용자가 결과 행동에 대해 불확실해지고 전환율이 감소합니다.

## UX 라이팅 가이드

토스 앱의 보이스톤을 적용한 문구 작성 지침입니다.

### 1. 해요체 사용

**핵심 원칙:**
- 모든 제품 내 문구는 반드시 "해요체"로 작성
- 상황과 맥락을 불문하고 일관되게 적용
- 친근하고 캐주얼한 느낌 전달

**예시:**
- ❌ "결제가 완료되었습니다"
- ✅ "결제가 완료됐어요"

### 2. 능동적 말하기

**변환 규칙:**

| 금지 | 권장 |
|------|------|
| ~됐어요 | ~했어요 |
| ~었 (과거형 어미) | 현재형 |
| 수동형 문장 | 능동형 문장 |

**예시:**
- ❌ "포인트가 적립됐어요"
- ✅ "포인트를 적립했어요"

### 3. 긍정적 말하기

**변환 규칙:**

| 금지 | 권장 |
|------|------|
| 없어요 | 있어요 |
| 안 돼요 | 할 수 있어요 |
| 불가능해요 | 가능해요 |

**에러 메시지:**
- 오류 상황에서도 건설적 안내 제공
- 해결 방안 제시

**혜택 제한 안내:**
- "받을 수 없을 때"도 긍정형 표현
- 서비스 이용 가능성 강조

### 4. 캐주얼한 경어

**제거 대상:**

| 금지 | 권장 |
|------|------|
| ~시겠어요? | ~할래요? |
| 계세요 | 있어요 |
| 여쭈다 | 확인하다, 묻다 |
| ~께 | ~에게 |

**예시:**
- ❌ "결제하시겠어요?"
- ✅ "결제할래요?"

### 5. 명사+명사 사용 금지

**변환 규칙:**
- 한자어를 동사 형태로 변환
- "{명사}가 {명사}해서" 형태 활용

**예시:**
- ❌ "결제 완료"
- ✅ "결제가 완료됐어요"

### 예외 규칙

**수동형 허용:**
- 서비스 종료, 기간 만료
- 사용자에게 미치는 영향 (연체, 해지)
- 사용자 안심 메시지

**경어 허용:**
- 사용자 맥락 활용 질문 ("~시나요?")
- 사용자 상황 추정
- 선의 필요 상황 (설문조사)

**부정형 허용:**
- 서비스 정책상 사용 불가
- 일부 기능만 사용 불가
- 민감 정보 처리 안내

### 핵심 정리

| 원칙 | 금지 | 권장 |
|------|------|------|
| 문체 | 존댓말, 하십시오체 | 해요체 |
| 목소리 | 수동형, ~었어요 | 능동형, 현재형 |
| 톤 | 부정, 없어요, 안 돼요 | 긍정, 있어요, 할 수 있어요 |
| 경어 | ~시겠어요, ~께 | 캐주얼, ~에게 |
| 어휘 | {명사}+{명사} | 동사화, 풀어쓰기 |

## 그래픽 가이드

### 저작권 안내

토스의 모든 그래픽 자산은 ㈜비바리퍼블리카의 지식재산권입니다.

**사용 범위:**
- 앱인토스 제휴 환경 내 서비스 운영 및 홍보 목적만 허용
- 다른 서비스나 매체에서의 복제·수정·배포·전송·공중송신 금지

### 제공 리소스

#### 1. 아이콘 & 이모지

**제공량:** 약 7,000개 이상

**사용 지침:**
- 화면에서 24~40px 사이의 크기로 사용
- 아이콘 두 개 이상 병렬 조합 지양
- 한 번에 하나씩만 사용

**커스텀 제작:**
- 아이콘 제작 가이드라인 참고 필요

#### 2. 스마트폰 목업 파일

**제공 크기:**
- Full
- Large (Top/Bottom)
- Medium (Top/Bottom)
- Small (Top/Bottom)

**사용 지침:**
- 제공 파일 그대로 사용 권장
- 임의 크롭, 색상 보정, 왜곡 금지
- 개발 시 URL 직접 삽입 권장 (디자인 업데이트 자동 반영)

#### 3. 토스트 (AI 이미지 생성 도구)

- 아이콘 및 이모지 기반 3D 이미지 생성
- 생성된 그래픽은 토스 그래픽 디자인 팀 승인 필요 (1일 이내)

#### 4. 기타 그래픽

- 3D, 애니메이션 등은 제공된 모듈 내 포함 자산만 사용 가능

### 올바른 사용 지침

1. **맥락에 맞는 그래픽 사용**
   - 그래픽은 장식이 아닌 사용자 이해 증진 용도

2. **디테일에 맞는 크기**
   - 단순 그래픽은 작게
   - 복잡한 그래픽은 크게

3. **그래픽 수량 제한**
   - 한 화면에 비슷한 크기의 그래픽이 많으면 시선 분산
   - 핵심 그래픽 하나 + 보조 그래픽(아이콘)으로 강조

4. **배치 고려**
   - 중요한 정보가 가려지지 않도록 크기와 위치 조정

5. **부정적 감정 표현 지양**
   - 부정적이거나 호소성 감정 표현은 다크 패턴

6. **장식 효과 지양**
   - 의미없는 묘사, 파티클, 그라데이션 등 지나친 장식 금지

7. **정확한 상황 그래픽 사용**
   - 오류 아닐 때 느낌표 아이콘 사용 금지
   - 대기 불필요할 때 로딩 애니메이션 사용 금지

### 파트너 제작 그래픽 가이드

1. **토스 스타일 일관성 유지**
   - 단순 명료하고 깨끗한 디지털 그래픽 스타일
   - 손그림, 서정적 화풍, 만화 표현 지양

2. **고품질 그래픽**
   - 깨끗하고 선명한 고화질 제작
   - 열화 및 파티클 효과 지양

3. **다크/라이트 모드 호환성**
   - 양쪽 모드에서 가시성 확보
   - 중간 명도 그래픽 권장

4. **색상 및 레이아웃 조화**
   - 텍스트, CTA 버튼 등 화면 요소와 균형

5. **긍정적 표현**
   - 서비스의 안정성과 신뢰감 전달
   - 모노톤 과다 사용, 뿌옇거나 칙칙한 디자인 지양

## 해상도 가이드

다양한 기기 환경을 하나의 논리 해상도로 대응합니다.

### 핵심 설계 철학

하나의 논리 해상도를 기준으로 설계하고, 에셋은 1x·2x 그룹으로 준비하면 앱인토스 실제 트래픽의 대부분을 안정적으로 커버할 수 있습니다.

### 전체 화면 요구사항

미니앱은 전체 화면 기능을 구현해야 합니다:

- 콘텐츠가 디스플레이 전체를 채움
- 웹뷰 여백이나 반투명 영역 없음
- 가로세로비 왜곡이나 레터박스 없이 회전 처리
- 노치, 카메라 홀, 다이나믹 아일랜드 등 Safe Area 처리
- 스케일링 열화 방지

### 두 가지 해상도 프레임워크

**논리 해상도 (Logical Resolution)**
- UI 포지셔닝, 좌표 계산, 게임 로직 기준선
- 실제 디바이스 픽셀 해상도와 분리
- 모든 기기에서 단일 선택 권장

**에셋 해상도 (Asset Resolution)**
- 이미지, 배경, 캐릭터, UI 리소스 픽셀 밀도
- 기기별 에셋 준비 불필요
- 그룹화로 관리

### 권장 사양

**논리 해상도 범위:**
- 세로: 약 360 × 640 ~ 420 × 740
- 가로: 약 640 × 360 ~ 740 × 420

**에셋 해상도 그룹:**
- 1x 에셋: 기준선 커버리지
- 2x 에셋: 고해상도 기기 지원
- 3x 에셋: 그래픽 중요 앱용 선택 사항

### 데이터 기반 분석

트래픽 집중도 분석 결과:
- 미니앱 트래픽의 약 70-80%가 특정 뷰포트 범위에 집중
- 세로: 360-420px 너비
- 가로: 800-900px 너비

나머지 트래픽은 개별 최적화 불필요한 다양한 해상도에 분포

### 테스트 기기 요구사항

3~5개의 대표 기기로 충분:
- 다양한 화면 비율 (2-3개 기기)
- Safe Area 구현 기기 (1개)
- 작은 화면 크기 기기 (1개)

### 피해야 할 관행

- 기기 모델별 논리 해상도 변경
- 실제 픽셀 해상도를 UI/좌표 기준으로 사용
- 과도한 에셋 해상도 그룹 증가
- 해상도 의존적 레이아웃 변경

이러한 접근은 UI 열화, 레터박스, 유지보수 비용 증가를 초래합니다.

## 피그마 디자인 가이드

### Step 1: 피그마 접근 권한 요청

채널톡을 통해 피그마 권한을 요청합니다. 승인되면 TDS 라이브러리가 있는 초대된 프로젝트에 접근할 수 있습니다.

### Step 2: TDS 컴포넌트 접근

두 가지 메인 라이브러리:

- **TDS Components for AppsInToss**: 다양한 UI 컴포넌트
- **TDS Foundation for AppsInToss**: 폰트, 색상, 아이콘 라이브러리

키보드 단축키 `option + 2` 또는 왼쪽 Assets 패널로 컴포넌트 검색

### 컴포넌트 상태 표시

색상 원으로 구분:

| 색상 | 의미 |
|------|------|
| 🔵 파란색 | 개발 문서에 코드 제공된 TDS 컴포넌트 |
| 🟣 보라색 | 피그마에서 사용 가능, 코드 미제공 (커스텀 개발 필요) |
| 🔴 빨간색 | 디자인 커스터마이징 제한된 TDS 컴포넌트 |
| 🌈 무지개색 | 개발 문서 없는 컴포넌트 (커스텀 코드 필요) |

### Step 3: 에셋 및 이미지 사용

**아이콘 관리:**
- 제공된 아이콘 검색 후 대체
- 커스텀 아이콘은 가이드라인 따라 제작 후 컴포넌트로 변환

**이미지 처리:**
- 빈 이미지 플레이스홀더에 `Cmd + Click`
- 원하는 이미지 직접 업로드

### 디자인 베스트 프랙티스

1. **네비게이션**
   - 모든 화면 상단에 🔴 Mobile_Navigation 배치
   - 🌈 Screen 템플릿 사용 시 필수 컴포넌트 자동 포함

2. **텍스트 요소**
   - 피그마 텍스트 대신 🔵 Mobile_Text 컴포넌트 사용
   - 플랫폼 간 큰 텍스트 자동 지원
   - 일관된 폰트 크기, 자동 라인 높이/자간 처리

3. **속성 편집**
   - 오른쪽 패널에서만 속성 수정
   - 캔버스 직접 조작은 문서화되지 않은 속성 생성

4. **화면 크기**
   - 가능하면 375px 너비로 작업
   - 일부 컴포넌트(🔵 Mobile_Keypad)는 다른 크기에서 반응형 지원 부족

5. **공통 레이아웃**
   - 대부분 화면은 🔵 Mobile_Top + 🔵 Mobile_ListRow로 효과적 구성

### 간격 및 패딩

대부분 TDS 컴포넌트는 좌우 패딩 내장:
- 갭 없이 컴포넌트 바로 연결 가능
- 🔵 Mobile_ListRow는 4가지 패딩 옵션: narrow, standard, wide, extra-wide

추가 간격 필요 시:
- autolayout gap 사용
- 🌈 Padding・Margin 컴포넌트 (프리셋 마진 값)

### 문제 해결

**컴포넌트 작동 안 함:** 개발자 커뮤니티에 문의

**컴포넌트 없음:** 기존 TDS 컴포넌트로 대체 가능한지 확인 (예: 커스텀 카드 UI 대신 🔵 Mobile_ListRow)

**폰트 설치:** 컴포넌트에 미리 적용됨, 별도 설치 불필요

**다크 모드:** 현재 미지원, 라이트 모드 전용 디자인

## 앱빌더 (App Builder) 가이드

### 1. 앱빌더 소개

앱인토스 파트너를 위한 UI 디자인 도구입니다.

**주요 장점:**
- 디자인 라이브러리 없이 필요한 컴포넌트 내장
- 오른쪽 패널에서 색상, 텍스트, 간격 쉽게 조정
- 웹 기반으로 설치 불필요
- 콘솔에서 접근: 미니앱 선택 → 디자인 메뉴

**현재 상태:**
- 베타 버전 (기능 누락이나 오류 가능)
- 기본 UI 디자인 기능만 제공
- 스크린샷이나 화면 녹화로 외부 공유 금지

### 2. 프로젝트 관리

**프로젝트 생성:** 홈 화면 우측 상단 "+ 새로 만들기" 버튼

**프로젝트 삭제:** 영구 삭제 불가, 보관 및 복원만 가능

**즐겨찾기:** 프로젝트 호버 후 별 아이콘 클릭

**프로젝트 폴더:** 왼쪽 패널 "+" 버튼으로 생성

### 3. 디자인 시작하기

**브랜드 스타일 설정:**
- 새 프로젝트 열면 "시작하기" 모달 표시
- 서비스 이름: 앱인토스 콘솔에 등록된 이름만 선택 가능
- 버튼 색상: 대부분 에셋에 적용되는 프라이머리 브랜드 색상

**페이지:**
- 준비하기 page: 기본 앱빌더 사용 설명
- 템플릿 page: 바로 사용 가능한 UI 세트

**디자인 방식:**
1. Quick Start: 기존 토스 UI 화면에서 옵션 수정
2. Custom: 디자인 에셋 조합으로 완성

### 4. Quick Start 디자인

토스 앱의 UI 화면을 가져와 제한된 옵션 수정이 가능합니다.

**수정 방법:**
- 오른쪽 패널에서 텍스트 및 설정 변경
- `Ctrl(Cmd) + click`으로 내부 요소 선택
- 오른쪽 패널 옵션 외의 요소 수정/삭제 불가

### 5. Custom 디자인

토스 디자인 에셋을 조합해 UI를 구축합니다.

**베이스 화면 추가:**
- 화면 크기는 iPhone 13 mini 기준 (375x812)
- 화면 너비 임의 조정 금지

### 6. 텍스트

TDS 타이포그래피를 효율적으로 적용합니다.

**텍스트 유형:**
- 일반형: 표준 멀티라인 형식
- 포스트형: 헤더에 적합한 볼드 형식

**텍스트 편집:**
- 텍스트 입력 필드에서 내용 편집
- 줄바꿈은 Shift+Enter

### 7. 아이콘 및 그래픽

**토스 팀이 제공한 그래픽만 사용 가능합니다.**

**그래픽 추가 방법:**
1. 상단 컨트롤 바 'Resources' 버튼 → 그래픽 선택
2. 오른쪽 패널 Custom 섹션: 'Asset' 추가 → 'Change' 버튼 → 그래픽 선택

**3D 그래픽:**
- 앱빌더 베타에는 내장 3D 그래픽 미제공
- 2D 그래픽을 3D로 변환하는 대안 사용

### 8. 레이아웃

**그룹화 단계:**
1. Shift+click으로 여러 에셋 선택
2. 나타나는 툴바에서 수평/수직/겹침 선택
3. detach로 그룹 해제

**스택 레이아웃 옵션:**
- Fit: 콘텐츠에 맞게 길이 조정
- Fill: 화면 너비까지 확장
- Fixed: 사용자 지정 길이
- Distribute: 배치 시작점
- Gap: 요소 간 간격
- Padding: 가장자리 여백

**스타일 옵션:**
- Visible: 표시/숨김 토글
- Opacity: 투명도 (1 = 100%)
- Fill: 배경색
- Border: 테두리 색
- Radius: 모서리 둥글기
- Shadow: 그림자 효과

### 9. 프로토타입

**재생 버튼 (우측 상단):** 모바일 해상도로 프로토타입 표시

**추가 기능:**
- 상단 컨트롤 바 폰 아이콘으로 기기 미리보기
- 다양한 기기에서 해상도 확인
- 반응형 디자인을 위해 모든 에셋 너비를 "Fill"로 설정
- 프로토타입 공유 버튼으로 모바일 기기 보기용 링크 복사

## TDS (Toss Design System)

### 개요

TDS는 토스 커뮤니티 전반에서 사용되는 포괄적 디자인 시스템입니다.

**핵심 가치:**
- 부서 간 협업을 가능하게 하는 디자인 언어 및 개발자 도구
- 커스텀 UI 대비 3-5배 빠른 개발 완료

### 핵심 컴포넌트 라이브러리

자주 사용되는 11개 컴포넌트:

| 컴포넌트 | 설명 |
|----------|------|
| Badge | 시각적 지표 및 라벨 |
| Border | 경계 및 구분선 요소 |
| BottomCTA | 화면 하단 CTA 배치 |
| Button | 인터랙티브 터치 타겟 |
| Asset | 공통 시각 리소스 |
| ListRow | 개별 리스트 아이템 컨테이너 |
| ListHeader | 리스트 섹션 헤더 |
| Navigation | 화면 네비게이션 컨트롤 |
| Paragraph | 텍스트 콘텐츠 컴포넌트 |
| Tab | 탭 네비게이션 인터페이스 |
| Top | 상단 네비게이션 요소 |

### 사용 권한 및 법적 프레임워크

**지식재산권:**
- 앱인토스를 통해 제공되는 모든 자료는 토스 소유

**허용 사용 범위:**
- 파트너는 앱인토스 서비스 제공 범위 내에서만 사용 가능
- 소유권 이전이 아닌 제한적 라이선스

**준수 요구사항:**
- 약관 및 규정 준수 필수
- 위반 시 서비스 종료 또는 추가 조치 가능

### 구현 리소스

- **디자인 도구**: 피그마 통합, Deus (앱빌더) 지원
- **개발 레퍼런스**: WebView 및 React Native 튜토리얼
- **외부 문서**: Frontend Fundamentals 접근성 가이드라인
- **코드 예제**: Bedrock SDK 레퍼런스 자료

### 접근성 표준

Frontend Fundamentals 외부 문서를 통해 제공되는 전용 접근성 가이드라인 포함
